# 要件定義書

## プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | zuncha（ずんちゃ） |
| 概要 | ずんだもんキャラクターとリアルタイム音声会話ができるWebアプリケーション |
| 目的 | 音声入力・AI応答・音声合成・表情アニメーションを統合し、ずんだもんと自然な会話体験を提供する |
| 策定日 | 2026-06-09 |
| 最終更新 | 2026-06-15 |

---

## 機能要件

### 1. 音声入力（STT）

| ID | 要件 |
|----|------|
| F-STT-01 | ユーザーのマイク音声をテキストへリアルタイム変換する |
| F-STT-02 | 音声認識失敗時にずんだもんがリトライを促すメッセージを発話する |
| F-STT-03 | 一定時間無音・認識不能が続いた場合にタイムアウト処理を行う |
| F-STT-04 | 騒音環境下での認識精度を考慮した実装とする |

### 2. AI応答生成

| ID | 要件 |
|----|------|
| F-AI-01 | ユーザーの発話テキストをもとにずんだもんらしい応答を生成する |
| F-AI-02 | 応答生成と同時に感情ラベルを出力する |
| F-AI-03 | 同一会話ID内の直近の会話履歴をコンテキストとして参照して応答を生成する |

### 3. 感情分析・表情切り替え

| ID | 要件 |
|----|------|
| F-EMO-01 | AI応答の内容から感情状態をリアルタイムに分析する |
| F-EMO-02 | 対応感情: 喜び・怒り・悲しみ・楽しい・照れ・困惑・ドヤ顔（最低7種） |
| F-EMO-03 | 感情ラベルに対応したずんだもんの画像へリアルタイムで切り替える |
| F-EMO-04 | 感情切り替えはSSEで受信したイベントをトリガーとする |

### 4. 音声合成（TTS / VOICEVOX）

| ID | 要件 |
|----|------|
| F-TTS-01 | AI応答テキストをVOICEVOXでずんだもんボイスに変換して再生する |
| F-TTS-02 | VOICEVOXはAPIラッパー層を介して呼び出す（直接依存を排除） |
| F-TTS-03 | ラッパー層の差し替えにより代替TTSエンジンへ移行できる設計とする |

### 5. リアルタイム通信

| ID | 要件 |
|----|------|
| F-RT-01 | サーバーからクライアントへのイベント（応答テキスト・感情ラベル・音声データURL等）をSSEで配信する |
| F-RT-02 | 接続断時に自動再接続を行う |

### 6. 会話履歴

| ID | 要件 |
|----|------|
| F-HIST-01 | 会話セッション開始時に一意なID（ULID）を自動採番する |
| F-HIST-02 | 画面上に全ユーザーの会話履歴一覧を表示する（開始日時・冒頭テキスト等）※全員の会話が共有される仕様 |
| F-HIST-03 | 一覧から会話を選択して過去の内容を閲覧できる |
| F-HIST-04 | 会話のURLに会話IDを含める（例: `/c/{conversationID}`）。会話IDはURL共有のための内部管理キーとして機能する |
| F-HIST-05 | 会話一覧の表示項目: 開始日時・最初のユーザー発話の冒頭テキスト（上限文字数は実装時に決定） |
| F-HIST-06 | 会話の手動削除機能は提供しない。削除はF-HIST-07の自動削除のみとする |
| F-HIST-07 | 会話開始から30日経過したセッションを自動削除対象とし、新しい会話開始時にチェック・削除を実行する |

### 7. 認証・ユーザー管理

| ID | 要件 |
|----|------|
| F-AUTH-01 | アカウント作成不要・ゲストモードのみで即時利用できる |

## 非機能要件

### パフォーマンス

| ID | 要件 |
|----|------|
| NF-PERF-01 | 音声入力からAI応答音声再生開始までの遅延を3秒以内とすることを目標とする |
| NF-PERF-02 | Whisper.cpp はGPUなしのCPU動作で許容遅延範囲内に収まることを確認する |
| NF-PERF-03 | VOICEVOXとSTTの同時処理時にサーバーが過負荷にならない設計とする |

### スケーラビリティ

| ID | 要件 |
|----|------|
| NF-SCALE-01 | 同時接続数は最大10人を想定した設計とする（社内利用） |
| NF-SCALE-02 | 単一インスタンスで動作可能な設計とする（水平スケールは初期スコープ外） |

### セキュリティ

| ID | 要件 |
|----|------|
| NF-SEC-01 | ユーザー入力は全てバリデーション・サニタイズを実施する |
| NF-SEC-02 | APIエンドポイントには社内ドメインのみを許可するCORS設定を施す |

### プライバシー

| ID | 要件 |
|----|------|
| NF-PRIV-01 | 会話データの保持期間（30日）を社内向けに明示する |
| NF-PRIV-02 | 30日経過した会話データは自動削除する（F-HIST-07と連動） |

### 可用性・保守性

| ID | 要件 |
|----|------|
| NF-MAINT-01 | VOICEVOXラッパー層により、TTS依存を局所化して交換コストを最小化する |
| NF-MAINT-02 | 開発・本番環境をDocker Composeで統一し、環境差異を排除する |

## 技術スタック

| 領域 | 採用技術 | 備考 |
|------|---------|------|
| フロントエンド | SvelteKit（TypeScript） | |
| バックエンド | Go | |
| データベース | PostgreSQL | Docker Compose で管理、開発・本番統一 |

| 音声合成（TTS） | VOICEVOX（APIラッパー層経由） | 差し替え可能設計 |
| インフラ | Docker Compose | |
| 会話ID採番 | ULID | ソート可能・URL安全 |

## システム構成概要

```
[ブラウザ]
  |
  |--- マイク音声 ---> [バックエンド: Go]
  |                        |
  |                        |--- Whisper.cpp (ローカルSTT)
  |                        |--- AI応答生成 + 感情分析
  |                        |--- VOICEVOXラッパー (TTS)
  |                        |--- PostgreSQL (会話履歴保存・30日保持)
  |                        |
  |<-- SSE (応答テキスト・感情ラベル・音声URL) ---
  |
  |--- SvelteKit
        |--- 感情ラベルに応じたずんだもん画像切り替え
        |--- 音声再生
        |--- 会話UI（/c/{conversationID}）
        |--- 会話履歴一覧（全ユーザー共有）
```

---

## 制約・前提条件

| # | 内容 |
|---|------|
| C-01 | Whisper.cpp はサーバー側でローカル動作させる（クラウドSTT不使用） |
| C-02 | VOICEVOX は利用規約に従って使用する。規約変更・停止に備えてラッパー層で抽象化する |
| C-03 | 認証機能は実装しない。ゲストモードのみとする |
| C-04 | 開発環境はDocker Composeを前提とする |
| C-05 | PostgreSQL を開発・本番で統一し、SQLiteのような環境差異を生じさせない |
| C-06 | 会話履歴は全ユーザーに公開される（社内限定利用を前提とした設計） |
| C-07 | 会話データの保持期間は30日とし、新規会話開始時に期限切れデータを削除する |
