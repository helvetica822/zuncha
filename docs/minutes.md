## 議題

---

## 各ラウンドのサマリー

### ラウンド1

**ずんだもん（アイデアマン）**
- AIによる感情分析をリアルタイムで実施し、感情状態をずんだもんの画像に連動させる提案。
- 喜怒哀楽のみならず「照れ」「困惑」「ドヤ顔」など細かい感情の検出を想定。

**冥鳴ひまり（リスク提言）** — 悲観度 ★★★☆☆
- VOICEVOX APIの利用規約変更・サービス停止リスクを指摘。
- 代替TTSエンジンへの差し替えを見越したAPIラッパー層の設計を提案。

**春日部つむぎ（ユーザー目線）**
- STT（音声認識）のエラーハンドリング不足を懸念。
- 騒音環境での誤認識、認識失敗時のリトライ促進、タイムアウト処理を仕様に含めるよう要求。

---

### ラウンド2

**ずんだもん（アイデアマン）**
- DBにDynamoDB、リアルタイム通信にSSE（Server-Sent Events）、STTにWhisper.cpp（ローカル）を提案。
- クラウド依存ゼロ・低遅延・GPUなしCPU動作を訴求ポイントとして挙げた。

**冥鳴ひまり（リスク提言）** — 悲観度 ★★★★☆
- WebSocket採用時の接続数増大によるサーバーリソース枯渇リスクを指摘。
- VoiceVoxとSTTの同時処理による負荷集中を懸念。
- スケールアウト戦略を初期設計から組み込む必要性を主張。

**春日部つむぎ（ユーザー目線）**
- 会話履歴の保存範囲とプライバシーポリシーの未整備を問題提起。
- データ保持ポリシーの明文化を求めた。

---

### ラウンド3

**ずんだもん（アイデアマン）**
- DBをSQLiteに変更提案。Dockerローカル開発でファイル1個完結、本番移行時はPostgreSQLへ切り替え可能と主張。

**冥鳴ひまり（リスク提言）** — 悲観度 ★★★★☆
- ユーザー認証を「不要」のまま進めると、将来的な追加コストが膨大になると警告。
- 最初から認証前提の設計にしておくだけでリスクを大幅に低減できると主張。

**春日部つむぎ（ユーザー目線）**
- 認証なしは絶対NG、ただし重い会員登録はUX的に離脱を招くと指摘。
- ゲストモードで即利用可能にし、データ保存希望者のみアカウント作成する「段階認証」を提案。

---

## 決定事項一覧（めたん チームリーダー 最終決定）

| 論点 | 決定内容 |
|------|---------|
| フロントエンド | SvelteKit（TypeScript） |
| バックエンド | Go |
| DB | PostgreSQL（Docker Compose管理、開発・本番統一） |
| リアルタイム通信 | SSE（Server-Sent Events） |
| STT | Whisper.cpp（ローカル）＋ VOICEVOXラッパー層で抽象化 |
| VOICEVOX | APIラッパー層で抽象化（差し替え可能設計） |
| 表情切り替え | 感情分析連動（照れ・困惑・ドヤ顔など細かい感情） |
| 認証 | ゲストモード＋段階認証（データ保存希望者のみアカウント作成） |
| プライバシー | データ保持ポリシーを明文化・ユーザー向けに公開 |
| STTエラーハンドリング | リトライ促進・タイムアウト処理を必須仕様として実装 |
