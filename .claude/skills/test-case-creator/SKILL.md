# Test Case Creator

仕様書や要件定義から「何をテストするか」を網羅的に洗い出し、テストケース一覧を作成する。

**このスキルのスコープ**: テストケースの列挙まで。テストコードの実装はしない。

> **参照**: テスト設計の詳細ガイドラインは `.claude/rules/tdd-comprehensive.md` にある。

---

## ワークフロー

### Step 1: 仕様を読む

ユーザーから受け取った仕様を理解する。形式は何でもよい：
- Markdown 形式の仕様書（`requirements.md` のような要件定義）
- 機能ID単位の要件テーブル（例: `F-STT-01 | マイク音声をリアルタイムにテキスト変換する`）
- 機能の説明文
- 既存のコードファイル

不明な点があれば **1つだけ** 確認する。複数まとめて聞かない。

### Step 2: テスト対象の整理

仕様を読んで以下を自分で判断し、ユーザーに1-2行で共有する：

```
テスト対象: <モジュール/機能名>
主な責務: <1行で何をするものか>
```

### Step 3: テストケース一覧を出力する

以下の構造でテストケースを列挙する。

---

## テストケース一覧の出力形式

```markdown
## テストケース一覧: <対象名>

### 正常系

| # | テストケース名 | 入力条件 | 期待結果 | カバーする仕様 |
|---|-------------|---------|---------|-------------|
| 1 | <振る舞いを動詞で表現> | <具体的な入力値> | <具体的な期待値> | F-XXX-01 |

### 異常系・エラー系

| # | テストケース名 | 入力条件 | 期待結果 | カバーする仕様 |
|---|-------------|---------|---------|-------------|

### エッジケース

| # | テストケース名 | 入力条件 | 期待結果 | 備考 |
|---|-------------|---------|---------|-----|

### 未カバーの仕様

- <カバーできなかった仕様IDと理由>
```

---

## テストケース名の書き方

テストケース名は **「〜のとき、〜になる」** の形で書く。実装の詳細ではなく振る舞いを表現する。

```
❌ 悪い例: useState で isLoading が true になる
✅ 良い例: 送信中はローディング表示になる

❌ 悪い例: API を呼ぶ
✅ 良い例: 有効な入力で送信すると注文が作成される
```

---

## エッジケースの網羅チェック

テストケースを列挙したあと、以下が含まれているか自己チェックする。足りないものがあれば追加する。

**値のエッジケース**
- [ ] 空値: `""` / `null` / `undefined` / `0` / `[]` / `{}`
- [ ] 境界値: 最小・最大・その前後（文字数制限、数値範囲など）
- [ ] 非常に長い値（文字列1000文字、配列10000件など）

**状態のエッジケース**
- [ ] 矛盾する状態の組み合わせ（`ローディング中 + 無効化`、`エラー + 成功` など）
- [ ] 連続操作・二重送信

**外部依存のエッジケース**
- [ ] 外部API/サービスの失敗・タイムアウト
- [ ] ネットワーク切断・再接続

---

## 出力後のアクション

テストケース一覧を出力したら、最後に一言添える：

> 「テストコードの実装は `/test-code-generator` または Playwright スキルで対応できます。」
